# ПР №9. Лицензирование и open-source compliance

## 1. Выбор лицензии
**Выбранная лицензия:** MIT
**Тип:** разрешительная (permissive)

**Что разрешает:** использовать, копировать, модифицировать, распространять, продавать, включать в закрытые продукты.
**Что требует:** сохранить уведомление об авторском праве и текст лицензии во всех копиях.
**Может ли кто-то продавать ваш код:** ДА, может — это прямо разрешено.

## 2. Результат licensee detect
Команда: `licensee detect .`
**Match %:** 100% — означает степень совпадения текста файла LICENSE с эталонным текстом лицензии MIT по алгоритму Sørensen–Dice. 100% — точное совпадение.
**Откуда GitHub знает лицензию:** GitHub использует библиотеку *licensee* (та же самая) — она сканирует репозиторий при push и сравнивает файл LICENSE с базой эталонных текстов через SPDX.

## 3. SPDX-заголовки
Пример заголовка:
```
SPDX-FileCopyrightText: 2026 Egor-Ladygin-IBAS-24-11-1 <minecraft01082002@gmail.com>
SPDX-License-Identifier: MIT
```
- **SPDX-FileCopyrightText** — кто и когда владеет авторским правом на этот конкретный файл.
- **SPDX-License-Identifier** — под какой лицензией (SPDX-идентификатор) распространяется файл.

**reuse lint:** проверяет весь репозиторий на соответствие спецификации REUSE (каждый файл должен иметь заголовок + все тексты лицензий в LICENSES/).

## 4. Лицензии зависимостей
| Пакет  | Лицензия        |
|--------|-----------------|
| requests | Apache-2.0    |
| flask    | BSD-3-Clause  |
| numpy    | BSD-3-Clause  |
| werkzeug | BSD-3-Clause  |

**Есть ли GPL/AGPL:** НЕТ. Все зависимости — permissive (BSD/Apache/MIT).
**Что было бы если бы был GPL:** GPL — копилефтная лицензия. При линковке с GPL-кодом весь ваш проект обязан стать GPL и открыть исходники. Для закрытого коммерческого продукта это **неприемлемо**. AGPL ещё жёстче — требует раскрытия исходников даже при использовании по сети (SaaS).

## 6. Анализ совместимости

**Сценарий 1 (MIT-проект + GPL-3.0 библиотека в закрытый продукт):**
❌ НЕЛЬЗЯ. GPL требует, чтобы производное произведение распространялось под GPL. Включив GPL-библиотеку в закрытый продукт, вы обязаны открыть весь свой исходный код.

**Сценарий 2 (Apache-2.0 проект + MIT библиотека):**
✅ МОЖНО. MIT совместима с Apache-2.0 в обе стороны. Apache-2.0 — более строгая, но допускает включение MIT-кода.

**Сценарий 3 (MIT-код продают без указания авторства):**
❌ НАРУШАЕТ. MIT прямо требует: *"The above copyright notice and this permission notice shall be included in all copies"*. Убрать авторство = нарушение лицензии.

**Сценарий 4 (AGPL-3.0 как backend SaaS без раскрытия исходников):**
❌ НЕЛЬЗЯ. AGPL специально создан, чтобы закрыть «лазейку SaaS»: если пользователи взаимодействуют с программой по сети — вы обязаны предоставить им исходный код. Для закрытого SaaS AGPL не подходит.

**Сценарий 5 (в репозитории нет LICENSE):**
⚠️ ПО УМОЛЧАНИЮ — **All Rights Reserved**. Код защищён авторским правом, использовать/копировать/модифицировать его **нельзя** без явного разрешения автора. Даже если код публичен на GitHub.

|--------|--------------|----------------------------|

## 8. Контрольные вопросы

**1. Чем разрешительная лицензия (MIT) отличается от копилефтной (GPL)? Что значит "вирусный эффект" GPL?**

Разрешительные лицензии (MIT, BSD, Apache) позволяют использовать код практически без ограничений, включая включение в закрытые продукты. Единственное требование — сохранить уведомление об авторском праве.

Копилефтные лицензии (GPL, AGPL) требуют, чтобы любые производные произведения распространялись под той же лицензией. "Вирусный эффект" GPL означает, что если вы включаете GPL-код в свой проект (даже как библиотеку), весь ваш проект обязан стать GPL и открыть исходный код. Это "заражает" весь код, контактирующий с GPL-библиотекой.

**2. Что такое SPDX-идентификатор? Зачем нужна машиночитаемая форма лицензии?**

SPDX (Software Package Data Exchange) — стандарт машиночитаемых идентификаторов лицензий (например, "MIT", "Apache-2.0", "GPL-3.0-only"). Машиночитаемая форма нужна для:
- Автоматической проверки compliance инструментами (reuse, licensee, pip-licenses)
- Интеграции в CI/CD пайплайны
- Анализа зависимостей и их лицензий
- Юридической определённости — однозначная идентификация версии лицензии

**3. Компания взяла open-source библиотеку под GPL-2.0 и встроила в закрытый коммерческий продукт. Что она нарушила и каковы последствия?**

Нарушение: GPL-2.0 требует, чтобы производные произведения (включая линковку с GPL-библиотекой) распространялись под GPL с открытым исходным кодом. Встраивание в закрытый продукт без раскрытия исходников — прямое нарушение лицензии.

Последствия:
- Судебный иск от правообладателей GPL-библиотеки
- Обязательство раскрыть исходный код всего продукта или удалить GPL-компонент
- Штрафы и компенсация убытков
- Репутационные потери
- В некоторых юрисдикциях — уголовная ответственность за нарушение авторских прав

**4. Что такое SBOM и зачем он нужен с точки зрения как compliance, так и безопасности?**

SBOM (Software Bill of Materials) — полный список компонентов ПО, их версий, лицензий и зависимостей. Аналог "списка ингредиентов" для программного обеспечения.

Для compliance:
- Отслеживание всех используемых лицензий
- Проверка совместимости лицензий
- Автоматическая проверка политик компании
- Доказательство due diligence при аудитах

Для безопасности:
- Быстрый поиск уязвимых компонентов (как в случае Log4Shell)
- Отслеживание версий зависимостей
- Выявление устаревших/неподдерживаемых библиотек
- Анализ цепочки поставок (supply chain attacks)

**5. Репозиторий на GitHub без LICENSE-файла. Можно ли использовать код из него в своём проекте? Обоснуйте.**

НЕТ, нельзя. По умолчанию действует режим "All Rights Reserved" — все права защищены. Публичная публикация на GitHub не означает передачу прав.

Обоснование:
- Согласно ст. 1270 ГК РФ и международному авторскому праву, исключительное право принадлежит автору
- Без явной лицензии использование = нарушение авторских прав
- GitHub Terms of Service разрешают только просмотр и форк (внутри GitHub), но не использование в других проектах
- Fork ≠ лицензия: форк — это техническая возможность GitHub, а не юридическое разрешение

**6. Чем отличается AGPL от обычной GPL? Какую "лазейку" она закрывает?**

AGPL (Affero GPL) — модификация GPL, закрывающая "SaaS-лазейку".

Обычная GPL требует раскрытия исходников только при **распространении** бинарников. Если вы используете GPL-код на сервере и предоставляете доступ через веб-интерфейс (SaaS) — это НЕ считается распространением, поэтому исходники можно не раскрывать.

AGPL добавляет требование: если пользователи взаимодействуют с программой **по сети**, вы обязаны предоставить им исходный код. Это закрывает лазейку для облачных сервисов, которые использовали GPL-код без раскрытия исходников.

**7. Зачем проверять лицензии зависимостей в CI, а не один раз при добавлении библиотеки?**

Причины для CI-проверки:
- **Транзитивные зависимости:** библиотека может обновиться и добавить GPL-зависимость в новой версии
- **Человеческий фактор:** разработчик может не заметить лицензию при добавлении
- **Автоматизация:** CI проверяет каждый PR/push автоматически
- **Историчность:** фиксация проверок для аудита и compliance
- **Масштабирование:** в проекте могут быть сотни зависимостей — ручная проверка невозможна
- **Непрерывность:** зависимости обновляются постоянно, разовая проверка устаревает через неделю
